在開發量化交易程式時，尤其是做「高頻交易（HFT）」或「Tick 級別的選擇權回測」，我們會面臨一個巨大的效能瓶頸：資料量太大了。

  以下是為什麼你需要編譯這個 cpp_engine 的具體原因與它的核心功能：

  1. 突破 Python 的效能瓶頸 (速度提升百倍以上)
   * 問題： Python 是一種直譯式語言（Interpreted Language）。當你要回放一整天的選擇權 Tick 資料時（可能高達幾十萬、幾百萬筆報價），如果用 Python 寫一個 for
     迴圈去一筆一筆讀取、更新委託簿（Order Book），然後每秒計算一次你的投資組合損益（Mark-to-Market, MTM），Python
     會跑得非常慢，一個月的回測可能要跑上好幾個小時。
   * C++ 的功能： C++ 是編譯式語言（Compiled Language），它的執行速度極快。我們把最耗時的「迴圈更新委託簿」跟「計算逐筆損益」的邏輯，用 C++ 寫在
     orderbook_replay.cpp 中。
   * 編譯的結果： 透過編譯，會把這個 C++ 程式碼打包成一個 .pyd 檔（相當於 Windows 的 DLL 檔）。這個檔案會透過 pybind11 這個橋樑，讓 Python 可以直接 import
     options_replay，就像呼叫一般的 Python 函式一樣簡單，但底層是用 C++ 的神速在跑。

  2. 精準重建委託簿狀態 (Order Book State Machine)
   * 功能： 當系統從 TimescaleDB 撈出一堆 Tick 資料（時間、成交價、最佳買價、最佳賣價）後，C++ 引擎的 SimulationEngine
     會負責把時間軸切成非常細緻的片段（例如毫秒級）。
   * 為何需要： 為了還原真實市場的搓合狀況，你必須知道在 09:15:33 那個瞬間，如果你丟一個「市價單」出去，你會買在哪個價位？C++ 引擎裡的 process_tick
     函數會不斷更新當時的「最佳買價 (Best Bid)」與「最佳賣價 (Best Ask)」。如果沒有這個引擎，你只能假設用「最新成交價 (Last
     Price)」成交，這在真實交易中會產生巨大的「滑價（Slippage）誤差」，導致回測失真（回測看起來賺錢，實盤卻虧錢）。

  3. 即時損益結算 (Mark-to-Market, MTM)
   * 功能： 在模擬回測的過程中，系統需要不斷知道「我現在到底賺多少、賠多少？是否該停損了？」。
   * C++ 的角色： OrderBookReplay::calculate_mtm(int position) 這個函數就是在做這件事。如果你做多（Buy Call），它會嚴格用當時的「最佳買價（Best
     Bid）」來計算你立刻平倉的價值（而不是用中間價或成交價），這種計算方式最貼近真實市場的流動性。因為它是 C++ 寫的，所以即使你每 1 毫秒查詢一次全市場幾十個合約的
     MTM，系統也不會卡頓。

  總結
  目前的系統中，有一個叫做 options_replay_fallback.py 的檔案，它是「純 Python 寫的降級備用引擎」。
   * 如果不編譯： 你的回測系統 simulator.py 會自動偵測，並改用這個純 Python 版本的引擎來跑。功能一樣，但速度會非常慢，適合用來除錯或開發策略。
   * 如果編譯了： simulator.py 會自動切換成 C++ 引擎。當你要跑半年、一年的大量資料做 AI 模型驗證時，這會幫你省下大量的等待時間（從幾小時縮短到幾分鐘）。
